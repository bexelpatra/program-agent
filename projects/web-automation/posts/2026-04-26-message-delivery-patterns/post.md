# 알림 시스템에서 메시지 미발송을 다루는 아키텍처 패턴 정리

## 문제 정의

외부 API(텔레그램, Slack, FCM 등)로 알림을 보내는 시스템은 언제든 발송에 실패할 수 있다. 네트워크 단절, DNS hiccup, 외부 API의 일시적 장애, rate limit, 그 외 OS 레벨의 일시적 라우팅 실패 등 원인은 다양하지만 공통점이 있다.

- **단발성 에러는 사후에 원인 특정이 거의 불가능하다.** 5초 뒤 재시도하면 멀쩡히 동작한다.
- 그러나 그 사이에 발송 못 한 메시지를 어떻게 처리하느냐에 따라 **사용자가 보는 상태의 일관성**이 결정된다.

특히 "상태 변화 알림"(어떤 자원이 사용 가능해졌다 / 마감됐다)을 다루는 시스템에서는, 메시지 한 통이 유실되는 순간 사용자 화면에 남아있는 정보가 실제 상태와 어긋나게 된다.

이 글은 그 미발송을 다루는 대표적인 아키텍처 패턴들을 정리하고, 각각의 트레이드오프를 비교한다.

---

## 전달 보장 의미론 (Delivery Semantics)

패턴들을 살피기 전에 용어부터 정리한다. 모든 메시징 시스템은 셋 중 하나로 분류된다.

| 의미론 | 보장 | 위험 |
|---|---|---|
| **At-most-once** | 메시지는 0번 또는 1번 전달됨 | 유실 가능 |
| **At-least-once** | 메시지는 1번 이상 전달됨 | 중복 가능 |
| **Exactly-once** | 정확히 1번 전달됨 | 구현 매우 어려움 (분산 환경에선 사실상 idempotency로 흉내 냄) |

대부분의 실용 시스템은 **At-least-once + 멱등성(Idempotency)** 조합으로 Exactly-once를 흉내 낸다. "보내고, 또 보내도 결과가 같게 만든다."

알림 시스템 관점에서 보면:
- 채팅 알림은 보통 At-most-once를 허용해도 큰 문제 없다 (사용자는 한 번 놓치면 다음 알림에서 캐치).
- 단, **상태 변화 알림**은 At-most-once가 곧 사용자 화면 일관성 깨짐으로 이어진다. 이 글의 본질적 문제 영역이다.

## 패턴 1. Retry with Exponential Backoff

가장 단순한 1차 방어선. 실패하면 잠깐 쉬었다가 다시 시도하고, 또 실패하면 더 길게 쉰다.

```
시도 1 → 실패 → 1초 대기
시도 2 → 실패 → 2초 대기
시도 3 → 실패 → 4초 대기
시도 4 → 실패 → 포기
```

대기 시간을 두 배씩 늘리는 이유는 단순하다. 방금 네트워크가 안 됐는데 0.01초 뒤에 다시 시도해봐야 같은 이유로 또 실패할 확률이 높다. 시간을 줘야 일시적 문제가 회복된다.

**Jitter**(무작위 지연 추가)를 곁들이면 더 좋다. 같은 시각에 실패한 N개의 클라이언트가 동시에 재시도하는 "thundering herd"를 막는다.

```python
delay = base * (2 ** attempt) + random.uniform(0, jitter_max)
```

### 장점
- 구현이 단순 (10~20줄).
- 단발성 hiccup의 90% 이상을 잡는다.

### 단점
- **호출 스레드를 점유한다.** 동기 코드에서 7~15초 retry를 다 소진하면 그만큼 cycle 처리 시간이 늘어난다.
- 외부 장애가 길어지면 모든 작업이 같이 느려진다.
- 재시도 중에도 결국 실패할 수 있고, 그땐 별도 처리 필요.

### 적합한 곳
- 짧고 빈번한 호출, 실패율이 낮은 환경.
- 사이클 시간 제약이 느슨한 곳.

## 패턴 2. Outbox Pattern

발송 시점과 처리 시점을 분리한다. 메시지를 외부 API로 직접 보내지 않고, **DB의 outbox 테이블에 INSERT**하고, 별도 워커가 그것을 읽어 발송한다.

```
[Cycle Worker]
  → INSERT INTO outbox (chat_id, payload, status='pending')
  → COMMIT (트랜잭션과 원자적 결합)

[Sender Worker (별도 프로세스/스레드)]
  → SELECT * FROM outbox WHERE status='pending' LIMIT N
  → 외부 API 호출
  → 성공 시 status='sent', 실패 시 retry_count++ 후 status='pending' 유지
```

핵심은 **DB 트랜잭션과 발송 의도가 원자적으로 묶인다**는 것. cycle worker는 "DB에 적었으니 내 책임은 끝"이고, 실제 발송은 sender가 자기 페이스로 처리한다.

### 장점
- Cycle 처리 시간이 외부 API 응답 시간과 무관해진다.
- 외부 API가 죽어도 cycle은 정상 동작.
- 재시도, 우선순위, rate limit 적용이 sender에 모두 위임된다.
- 메시지 유실이 사실상 0 (DB에 박힌 이상 언젠가는 발송됨).

### 단점
- 인프라 복잡도 증가 (sender 프로세스 추가, 모니터링 추가).
- 발송 지연(latency) 증가 가능 (sender의 폴링 간격만큼).
- 이미 무효해진 알림(상태가 다시 바뀜)을 그대로 보낼 위험. → "발송 직전 유효성 재확인" 단계 필요.

### 적합한 곳
- 처리량이 많은 시스템, 발송 신뢰성이 비즈니스에 중요한 곳.
- 결제 알림, 이메일 발송 등.

### 단순화 변형 — "DB를 큐로 쓰는 outbox"
별도 워커 없이, **다음 cycle에서 outbox를 함께 처리**하는 방식도 가능하다. 인프라는 그대로 두면서 outbox 패턴의 핵심(원자성)만 가져가는 절충안.

## 패턴 3. Dead Letter Queue (DLQ)

N회 재시도해도 실패한 메시지를 별도 큐(또는 테이블)로 격리한다.

```
[Sender]
  → retry_count >= MAX → DLQ로 이동, 정상 큐에서 제거
  → 운영자가 DLQ 검토 후 수동 재처리 / 폐기 결정
```

### 장점
- 영원히 재시도 루프에 빠지는 메시지를 막는다.
- 장애 분석 자료가 된다 (어떤 메시지가 왜 실패하는지).

### 단점
- DLQ 자체를 모니터링/관리하는 부담.
- 사용자 입장에선 결국 알림 못 받음.

### 적합한 곳
- Outbox 또는 정식 메시지 큐와 함께 운용되는 환경.

## 패턴 4. Circuit Breaker

연속 실패가 임계치를 넘으면 일정 시간 호출을 아예 중단한다.

```
상태 머신: CLOSED → OPEN → HALF_OPEN → CLOSED

CLOSED   : 정상. 호출 허용.
OPEN     : 차단. 모든 호출 즉시 실패 반환. 일정 시간 후 HALF_OPEN으로.
HALF_OPEN: 시험적으로 1개 호출 허용. 성공 시 CLOSED, 실패 시 OPEN.
```

외부 API가 죽어 있을 때 무의미한 호출로 자원 낭비/대기 시간 누적을 막는다.

### 장점
- 장애 외부 API에 cascading failure가 전파되는 걸 차단.
- 회복을 자동 감지.

### 단점
- 차단 중에는 메시지가 outbox나 DLQ로 쌓이게 되므로 다른 처리 메커니즘이 함께 필요.
- 임계치 튜닝이 까다로움.

### 적합한 곳
- 외부 의존도가 높고 장애 빈도가 있는 마이크로서비스 환경.

## 패턴 5. Idempotency Key

같은 작업을 여러 번 실행해도 결과가 동일하도록 보장한다. At-least-once 위에 idempotency를 얹어 사실상 Exactly-once 효과를 낸다.

```
client → request {id: "evt-65256", payload: ...}
server → 이미 evt-65256 처리됨? → 무시 / 동일 응답 반환
       → 처음 보는 id → 처리 후 기록
```

알림 시스템에서는 `event_id` 자체가 자연스러운 idempotency key가 된다. 같은 event_id에 대해 메시지가 중복 발송되지 않게 DB에 기록을 남긴다.

### 장점
- 재시도와 결합 시 안전하게 At-least-once를 쓸 수 있다.
- 분산 환경에서 가장 실용적인 Exactly-once 흉내.

### 단점
- 키 저장소 필요 (DB 또는 Redis).
- 키의 TTL 관리 필요.

## 패턴 6. Reconciliation (정합성 재확인)

이벤트 기반 알림과 별개로, **현재 상태(snapshot)를 주기적으로 비교**해서 누락을 보완한다.

```
사이클마다:
  1. 최신 snapshot 가져옴
  2. 직전 snapshot과 비교 → 변화분이 곧 이벤트
  3. 이벤트 발송
```

이 시스템(캠핑장 모니터)이 이미 사용하는 방식이다. 이벤트 자체가 "snapshot diff"이므로 메시지가 유실되어도 다음 사이클에서 새 변화가 생기면 그 시점부터 다시 정확한 상태를 받게 된다.

### 장점
- 메시지 유실에 대해 자연 회복력이 있다.
- 이벤트 누적 의존성이 없다 (언제든 처음 시작해도 됨).

### 단점
- "마지막 변화 이후 변화가 없다"면 유실은 영원히 묻힘.
- 폴링 주기와 외부 API rate limit의 트레이드오프.

### 적합한 곳
- 상태가 자주 바뀌는 모니터링 시스템.
- 이벤트 누락보다 "현재 상태"가 중요한 도메인.

## 패턴 7. Piggyback (편승 발송)

발송 실패한 메시지를 별도 재시도 없이 **다음번 발송에 묶어서** 함께 보낸다.

```
사이클 N:
  events = [A]
  send([A]) → 실패. alert_messages 테이블에 pending 마커만 남김.

사이클 N+k:
  new_events = [B]
  pending_events = [A]  ← 이전 실패분 조회
  send([A, B]) → 한 메시지에 합쳐 발송.
```

### 장점
- **정상 cycle 부담 거의 0.** 실패 안 났을 땐 SELECT 한 번만 추가, 결과 0행이면 즉시 종료.
- 별도 재시도 인프라 불필요.
- 사용자가 받는 알림 수도 줄어듦.

### 단점
- **다음 알림이 언제 올지 모른다.** 한가한 자원에선 유실분이 영원히 묻힐 수 있음.
- 그 사이 상태가 또 바뀌면 합쳐 보내는 정보가 무효해질 위험.
- 발송 직전 유효성 재확인 단계가 필요할 수 있음.

### 적합한 곳
- 이벤트 빈도가 높아 다음 발송이 곧 올 가능성이 큰 시스템.

## 패턴 8. Compensation Notification (보상 알림)

실패한 메시지의 내용을 복구하려 하지 않고, **"일관성이 깨졌다는 사실 자체"만 짧게 알린다.**

```
유실 마커 감지:
  → "⚠ 직전 알림 일부가 전송 실패했습니다. 화면 상태가 실제와 다를 수 있습니다."
  → 짧은 텍스트라 발송 부담 작음.
  → 발송 성공 시 마커 폐기.
```

### 장점
- 사용자가 일관성 깨진 상태를 인지하게 한다 → 직접 확인할 동기 부여.
- 구현 매우 단순 (~30줄).
- 자원 보유자가 한가한 자원이라도 일정 시간 내 통지 보장.

### 단점
- 어떤 메시지가 유실됐는지는 알리지 않음 (간단함 유지를 위해).
- 보상 알림 자체도 실패할 수 있음 (다음 사이클에서 자동 재시도).

### 적합한 곳
- 정확한 정보 복구보다 "사용자가 잘못된 상태를 믿고 있는 것을 막는 것"이 더 중요한 도메인.

## 패턴 9. 상태 기반 vs 이벤트 기반 알림

근본적으로 알림 모델 자체를 바꾸는 접근이다.

| 모델 | 동작 | 유실에 대한 태도 |
|---|---|---|
| **이벤트 기반 (Event-driven)** | "A가 B로 바뀌었다"는 차분(diff)을 보냄 | 한 번 놓치면 영영 놓침 |
| **상태 기반 (State-driven)** | "지금 상태는 X다"를 주기적으로 보냄 | 다음 알림에서 자동 복구 |

상태 기반은 알림이 곧 "현재 상태"이므로 유실에 강하다. 단점은 변화가 없어도 알림이 가서 시끄러울 수 있다는 것.

**하이브리드**: 평소에는 이벤트 기반, 일정 주기로 한 번씩 상태 알림 (heartbeat 알림)을 섞는 방식도 흔하다.

## 패턴 10. Acknowledgement-based Delivery

수신자(또는 외부 API)의 ACK를 받아야만 발송 완료로 처리한다.

```
send → 외부 API → ACK 응답 받음 → DB에 sent 마킹
                ↓
                ACK 없음 → 미완료 상태 유지 → 재시도 대상
```

대부분의 메시징 시스템(텔레그램 포함)이 HTTP 응답으로 이미 이 모델을 쓰고 있다. HTTP 200 응답 = ACK. 응답을 못 받으면 발송 성공인지 실패인지 모르므로 보수적으로 "실패로 간주, 재시도" 한다.

### 함정
- "ACK 안 옴"이 곧 "전달 실패"는 아니다. 외부 시스템엔 들어갔는데 응답만 못 받았을 수 있음.
- 그래서 재시도하면 중복 발송 위험. → idempotency key가 필요해진다.

## 패턴 11. Desired State Reconciliation (Diff-based Notification)

지금까지의 패턴들은 대부분 "이벤트 발송 흐름은 그대로 두고, 그 위에 안전망을 덧대는" 방식이었다. 마지막으로 소개할 이 패턴은 **알림 모델 자체를 바꾼다.**

핵심 아이디어는 한 줄이다.

> diff의 baseline을 "직전 snapshot"이 아니라 **"사용자에게 마지막으로 알린 상태"**로 바꾼다.

```
사이클마다 사용자별로:
  current  = 현재 snapshot에 subscription 필터 적용
  notified = DB의 last_notified_state 로드
  diff     = compute_diff(notified, current)
              - notified에 없고 current에 있음 → 새 자리 알림
              - notified에 있고 current에 없음 → 마감 알림
  if diff:
      send_message(diff)
      if 성공:
          notified_state := current   ← 발송 성공한 시점에만 갱신
      else:
          그대로 둠 → 다음 사이클이 동일 diff로 자동 재시도
```

이 모델은 분산 시스템에서 잘 알려진 패턴이다. **Kubernetes controller**가 desired state(YAML)와 actual state(cluster)를 비교해 reconciliation loop를 돌리는 것, **React**의 virtual DOM diff가 마지막 렌더 결과와 새 상태를 비교해 변경분만 반영하는 것, **rsync**가 source와 destination의 차이만 전송하는 것 — 모두 같은 철학이다.

### 자동으로 해결되는 것들

1. **발송 실패 자연 복구.** retry 큐, backoff, outbox 모두 불필요. 발송 성공할 때까지 diff가 그대로 있다.
2. **이벤트 기반의 본질적 한계 해소.** "변화 한 번뿐"이 아니라 "상태 차이가 있는 한 계속". Piggyback의 "다음 increase 안 오면 묻힘" 문제도 해소.
3. **외부 리셋에 강함.** 사용자가 메시지를 지웠거나, 봇이 재시작됐거나, 일부 데이터가 손상돼도 `last_notified_state`만 정확하면 자동 동기화.
4. **일관성이 강제됨.** 메시지가 "사용자가 본 진실의 한 축"이 된다. 어긋난 순간 자동 탐지.

### 단점 / 트레이드오프

- **per-subscription 상태 저장 비용.** 구독 수 × 상품 수 × 슬롯 수만큼 row 증가. 작은 규모에선 무시 가능.
- **알림 단위 재정의 필요.** "1개 이벤트 = 1개 메시지"에서 "1 사이클 diff = 1 메시지"로. 같은 사이클 내 묶음이 자연스러워짐.
- **마감 처리 정책 결정.** 마감을 새 메시지로 보낼지, 기존 메시지를 편집할지 — 사용자 경험에 직결되는 결정 포인트.
- **사이클 사이 단발 변화 손실.** 두 폴링 사이에 잠깐 생겼다 사라진 자리는 못 잡음. 단 이벤트 기반에서도 동일 한계.

### 패턴 6 (Reconciliation)과의 차이

이름이 비슷하지만 적용 위치가 다르다.

| 항목 | 패턴 6 (Reconciliation) | 패턴 11 (Desired State Reconciliation) |
|---|---|---|
| diff baseline | 직전 snapshot | 마지막으로 알린 상태 (`last_notified_state`) |
| 적용 위치 | 데이터 수집 단계 | 알림 발송 단계 |
| 발송 실패 시 | 다음 변화 발생할 때까지 묻힘 | 다음 사이클에 자동 재시도 |
| 사용자 화면 일관성 | 보장 안 됨 | 보장됨 |

대부분의 시스템은 패턴 6은 자연스럽게 쓰지만(폴링 + diff), 패턴 11은 의식적인 설계가 필요하다.

### 적합한 곳

- 외부 API의 일시 장애가 사용자 화면 일관성에 직접 영향을 미치는 시스템
- 상태 변화 알림 (자리 모니터링, 재고 알림, 가격 변동 등)
- 쿠버네티스/React/rsync 같은 동기화 시스템 일반

## 선택 가이드

**소규모 시스템, 단발 hiccup이 주된 실패 원인이라면**  
→ Retry + Backoff 단독으로 대부분 해결. Idempotency만 유의.

**메시지 유실이 비즈니스에 치명적이라면 (결제, 인증)**  
→ Outbox + Idempotency + DLQ. Sender는 별도 프로세스로.

**이벤트 빈도가 높은 모니터링/대시보드 알림**  
→ Reconciliation을 기본으로 깔고, Piggyback으로 유실분 자연 흡수. 한가한 시점에는 Compensation으로 인지 보장.

**외부 API가 자주 죽거나 느린 환경**  
→ Circuit Breaker로 cascading 차단 + Outbox로 메시지 보존.

**사용자 화면 일관성이 핵심 가치인 상태 알림**  
→ Desired State Reconciliation. 발송 실패 처리가 별도 메커니즘이 아니라 원래 흐름의 자연스러운 일부가 된다.

## 사례: 점진 마이그레이션 전략

이벤트 기반 시스템을 운영 중에 Desired State Reconciliation으로 전환하려면, 한 번에 갈아엎기보다 단계적 접근이 안전하다. 실제 캠핑장 자리 모니터 시스템에서 적용한 마이그레이션 흐름을 공유한다.

### 전제 조건
- 기존 시스템: snapshot diff → events → 이벤트 기반 발송 (이벤트 한 번 유실되면 다음 변화까지 묻힘).
- 사용자 경험은 최대한 유지 (마감 알림은 기존 메시지 편집 방식 그대로).
- 메시지 묶음 단위(상품별 분리)도 유지.

### 마이그레이션 단계

**1단계 — 데이터 모델 추가 (운영 무영향)**  
`subscription_notified_state` 테이블만 추가. 기존 코드 변경 없음. 운영에 영향 0.

**2단계 — `last_notified_state` 갱신 로직 추가 (그림자 모드)**  
기존 발송 성공 직후 새 테이블에 함께 기록. 신규 구독 시엔 첫 사이클에 현재 snapshot으로 침묵 초기화. 알림 흐름은 100% 동일, 데이터만 쌓임.

**3단계 — 검증/보정 단계 추가 (feature flag)**  
사이클 끝부분에 reconciliation 한 단계 삽입. 불일치 자리만 보정 알림 발송. `[보정]` prefix로 구분. 환경 변수로 on/off. 발송 실패 처리가 처음으로 자연 복구되기 시작하는 지점.

**4단계 — 운영 관찰 (1~2주)**  
보정 발생 빈도, 발송 실패 패턴, false positive 케이스 수집. 데이터 기반으로 다음 단계 가부 판단.

**5단계 — 전면 reconciliation 전환**  
알림 발송 흐름을 reconciliation diff 기반으로 재작성. 기존 "이벤트 매칭 → 발송" 분기 폐지. close 처리는 기존 편집 메커니즘 재사용. `events` 테이블은 디버깅/분석용으로 유지.

**6단계 — 정리**  
feature flag 및 보정 분기 코드 제거. 흐름 문서화.

### 이 전략의 장점

- **운영 안전성**: 1~2단계는 사용자 영향 0. 3단계도 feature flag로 즉시 롤백 가능.
- **데이터 기반 결정**: 4단계에서 실제 운영 데이터를 보고 5단계 전환 판단.
- **부분 채택**: 3단계까지만 운영해도 발송 실패 자연 복구라는 핵심 가치는 이미 달성됨. 5~6단계는 코드 단순화 측면이 더 강함.
- **사용자 경험 보존**: 마감 편집 방식 유지로 가시적 변화 거의 없음.

이런 식으로 단계마다 명확한 가치와 롤백 지점을 두면, 큰 모델 전환도 운영 중에 안전하게 진행할 수 있다. 핵심은 "한 단계 끝마다 시스템이 의미 있게 안정 상태에 도달해야 한다"는 것이다.

## 마무리

메시지 미발송을 완벽히 0으로 만드는 시스템은 없다. 핵심은 **유실의 비용**과 **방지/복구 메커니즘의 비용**을 도메인 특성에 맞게 저울질하는 것이다.

- 결제 알림이라면 Outbox에 인프라를 투자할 가치가 있다.
- 캠핑장 자리 알림이라면 다음 사이클에서 자연 복구되는 구조 + 인지 보장 한 줄이면 충분할 수 있다.
- 채팅이라면 At-most-once를 그냥 받아들이는 것도 합리적 선택이다.

"어디까지 보장할 것인가"는 기술이 아니라 제품 결정이다. 패턴은 그 결정을 구현하는 도구일 뿐이다.
