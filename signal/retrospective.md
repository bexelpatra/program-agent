# Retrospective

## 프로젝트 요약
- 이름: 네이버증권 금융 지표 크롤러
- 태스크 수: 6개 (첫 시도 성공: 6, 재시도 필요: 0)
- 세션 수: 1

## 잘 된 점
- 병렬 실행(TASK-002+003, TASK-005+006)으로 전체 소요 시간을 줄였다.
- 라이브 테스트까지 포함하여 15개 테스트 전부 첫 시도에 통과했다.
- 비동기 크롤링으로 0.27초에 12건 수집 달성.

## 문제점

### 1. coder-report.md 병렬 충돌
- 무엇: TASK-002와 TASK-003을 병렬로 실행했을 때, 두 Coder Agent가 같은 `signal/coder-report.md`에 결과를 기록한다. 나중에 끝나는 에이전트가 먼저 끝난 에이전트의 report를 덮어쓴다.
- 왜: schema.md에서 "report 파일은 매 태스크마다 전체를 덮어쓴다"로 규정되어 있고, 같은 에이전트 유형은 하나의 report 파일을 공유하기 때문.
- 영향: Manager가 먼저 끝난 에이전트의 report를 읽기 전에 덮어써지면, 결과를 확인할 수 없다. 이번에는 TASK-002가 먼저 끝나서 바로 확인했지만, 타이밍에 따라 놓칠 수 있다.

### 2. import 경로 비일관성
- 무엇: `database.py`와 `scraper.py`는 `from config import ...`로 직접 임포트하지만, 이는 src/가 sys.path에 있을 때만 동작한다. `main.py`에서 sys.path를 수동으로 추가해서 해결했지만, 테스트 파일에서도 동일한 작업이 필요했다.
- 왜: src/ 디렉토리가 Python 패키지(__init__.py)로 구성되지 않았기 때문.
- 영향: 테스트나 외부에서 모듈을 임포트할 때마다 sys.path 조작이 필요하다.

### 3. 세션 재개 프로토콜의 반복 실행
- 무엇: hook에 의해 사용자의 모든 메시지에 "세션 시작 감지" 알림이 트리거된다. 이미 세션을 재개하고 작업 중인데도 매번 signal 파일을 읽으라는 지시가 반복된다.
- 왜: hook이 메시지 단위로 작동하여, 실제 새 세션인지 기존 세션의 연속인지 구분하지 못함.
- 영향: 불필요한 파일 읽기와 상태 보고가 반복될 수 있다.

## 파이프라인 개선 제안

### 제안 1: 병렬 실행 시 report 분리
- 대상 파일: signal/schema.md, CLAUDE.md
- 현재: 같은 에이전트 유형은 하나의 report 파일을 공유 (coder-report.md)
- 제안: 병렬 실행 시 `coder-report-{TASK-ID}.md` 형태로 태스크별 report를 생성하거나, report 파일명에 태스크 ID를 포함하는 규칙을 추가한다. 예: `signal/coder-report-TASK-002.md`
- 이유: 병렬 실행의 이점을 안전하게 활용하기 위해 report 충돌을 구조적으로 방지해야 한다.

### 제안 2: src/ 패키지 구조 가이드 추가
- 대상 파일: agents/coder.md
- 현재: "src/ 디렉토리에 코드를 작성한다"만 명시
- 제안: Python 프로젝트의 경우 src/__init__.py 생성 또는 pyproject.toml 기반 패키지 구조를 권장하는 가이드를 추가한다. 최소한 "첫 번째 코드 파일 생성 시 src/__init__.py도 함께 생성한다" 규칙 추가.
- 이유: sys.path 수동 조작은 유지보수가 어렵고, 테스트/외부 도구와의 호환성이 떨어진다.

### 제안 3: 세션 재개 프로토콜 조건 정교화
- 대상 파일: CLAUDE.md (세션 재개 프로토콜 섹션)
- 현재: "새 세션이 시작되면 반드시 아래 순서로 현재 상태를 파악한다"
- 제안: "이미 동일 세션에서 상태 파악을 완료한 경우, 중복 실행하지 않는다. hook의 세션 시작 감지는 실제 새 대화 시작 시에만 적용한다"를 명시한다.
- 이유: 매 메시지마다 signal 파일을 다시 읽는 것은 비효율적이며, 이미 진행 중인 작업 흐름을 불필요하게 중단할 수 있다.
